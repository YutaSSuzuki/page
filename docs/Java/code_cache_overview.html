JVM Code Cacheの仕組み

1 Javaソースが実行されるまで

Javaのコンパイルには、アプリケーションを起動する前のコンパイルと、JVM上で実行中に行われる コンパイルの2段階がある。

  1. javacが.javaを、JVM用の命令であるJava bytecodeを含む.classへ変換する。
  2. JVMが.classをロードする。
  3. 起動直後は、インタープリタがbytecodeを1命令ずつ解釈して実行する。
  4. JVMが頻繁に実行されるメソッドやループを見つける。
  5. JITコンパイラが、そのbytecodeをCPUが直接実行できるネイティブコードへ変換する。
  6. 生成したネイティブコードをCode Cacheへ保存し、次回以降は再利用する。

Javaコードが実行される流れ

.java
  ↓ javac（アプリケーション起動前）
.class / Java bytecode
  ↓ JVMがロード
インタープリタ実行
  ↓ 頻繁に実行される処理を検出
JITコンパイル（アプリケーション実行中）
  ↓
ネイティブコードをCode Cacheへ保存

このため、.classを作るjavacと、実行中に性能を最適化するJITコンパイラは別のものである。

2 Code Cacheの役割

Code Cacheは、JITコンパイラが生成したネイティブコードやJVM内部コードを保存するJVM内の メモリ領域である。同じJavaメソッドを何度呼んでも、基本的にはCode Cacheにある同じ コンパイル済みコードを再利用する。呼び出し回数が増えるだけで異なるコードが無制限に増えるわけではない。

Code Cacheが不足すると、JVMは新しいネイティブコードを配置できず、JITコンパイルを停止する場合がある。 その後に初めて実行するメソッドはインタープリタ実行のままになり、性能へ影響する可能性がある。

検証アプリケーションではCode Cacheを圧迫するため、Byte Buddyで異なるクラスとexecute()メソッドを 大量に生成し、それぞれをJITコンパイル候補にした。

3 HotSpotとTiered Compilation

HotSpotはOpenJDKに含まれるJVM実装である。Tiered Compilationは、HotSpotがJavaメソッドを 一度だけコンパイルするのではなく、実行状況に合わせて段階的にコンパイル・最適化する仕組みである。

用語 意味
HotSpot Java bytecodeの実行、メモリ管理、JITコンパイルなどを担当するJVM実装
Tiered 段階的という意味。実行方式をLevel 0～4で切り替える
C1 コンパイル時間が短いJITコンパイラ。早期に実行速度を上げ、必要に応じて情報を収集する
C2 C1などが集めた実行情報を利用し、時間をかけて高度に最適化するJITコンパイラ

起動直後からすべてをC2で最適化すると、コンパイル自体に時間とCPUを使って起動が遅くなる。 そこで、まずインタープリタやC1で動かし、長く繰り返し使われる重要なメソッドだけをC2へ進める。

3.1 Tiered CompilationのLevel

Level 実行方式 Code Cacheとの関係
0 インタープリタ bytecodeを逐次実行する。対象メソッドのコンパイル済みコードはCode Cacheにない
1 C1、プロファイルなし 短時間で生成した単純なネイティブコードをCode Cacheへ格納する
2 C1、限定的なプロファイル 呼び出し回数やループ回数を限定的に収集するコードを含む
3 C1、完全なプロファイル C2最適化に使う分岐傾向や呼び出し先などの実行情報を収集する
4 C2、最適化 収集した情報を使った高度な最適化コードをCode Cacheへ格納する

代表的な遷移はLevel 0 → Level 3 → Level 4である。ただし、必ずこの順番になるわけではない。 JVMの判断によってLevel 1や2を経由したり、最適化の前提が崩れたときにdeoptimizationして 下位Levelやインタープリタへ戻ったりする。

3.2 Levelが上がる基準

LevelはJavaコードから直接指定するものではない。HotSpotのTiered Compilation policyが、主に次の情報を 使ってメソッドごとに判断する。

判断材料 内容
メソッド呼び出し回数 メソッド入口を通った回数。繰り返し呼ばれるメソッドはコンパイル候補になりやすい
backedge回数 ループ末尾から先頭へ戻った回数。1回の呼び出しでも大きなループなら候補になり得る
プロファイルの蓄積 分岐の偏り、呼び出し先、型など、C2が最適化判断に使う情報が十分か
C1/C2 compile queue コンパイル待ちが多い場合、Level 2を経由するなど遷移経路や閾値が調整される
メソッドの性質 小ささ、ループの有無、コンパイル可能か、最適化解除の有無など
JVMの状態 JITが有効か、Code Cacheに配置可能な空き領域があるか

「1,000回呼べば必ずLevel 4」のような固定基準ではない。閾値や遷移経路はJDK、JVM option、 コンパイラの混雑状況、メソッドの形によって変わる。warmupIterations=10000もJITコンパイルを促すための 入力であり、すべての生成メソッドが必ずLevel 4になることを保証する値ではない。

検証ではHotを事前warm-upしてLevel 4へ到達させ、ColdはCode Cache満杯後まで呼ばなかった。 満杯によってJITコンパイラが停止すると、ColdはLevel 0から上がれずインタープリタで実行される。
